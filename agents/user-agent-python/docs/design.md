# User Agent Python Detailed Design

## 1. Role

The Python User Agent is a demo Agent client. It discovers Service Agents through Discovery Node and performs signed trusted invocation.

It is a business Agent subject and uses `ansMetadata.subjectType = "agent"` in its DID Document.

## 2. Runtime

The module is a `uv` project:

```text
agents/user-agent-python/pyproject.toml
agents/user-agent-python/uv.lock
```

Command:

```text
uv run --project agents/user-agent-python openagentnet-user-agent
```

The current implementation uses Python standard HTTP client APIs plus `cryptography` for Ed25519 signing and verification.

## 3. Local Identity and Credentials

Representative local data:

```text
data/user-agent/did-document.json
data/user-agent/keys/keypair.json
data/user-agent/credentials/user-agent-registration.json
data/user-agent/credentials/credentials-index.json
data/user-agent/credentials/by-dimension/
```

The User Agent holds a local `UserAgentRegistrationCredential`, generated by the development identity generator and signed by the Registrar key.

The credential index supports multiple credentials by dimension, issuer, subject, and credential ID.

## 4. Discovery Flow

The User Agent:

1. queries Discovery with a capability tag
2. receives signed Discovery results
3. selects the first matching `AgentService`
4. fetches the Service Agent profile
5. builds a trusted invocation envelope
6. calls Service Agent `/agent/hello`
7. verifies the Service Agent response signature
8. prints a structured demo result

The current demo focuses on connectivity and trust verification rather than ranking.

## 5. Trusted Invocation Envelope

The User Agent sends:

- `type = OpenAgentNetTrustedInvocation`
- `callerDid`
- `targetDid`
- `nonce`
- `timestamp`
- request `body`
- `bodyHash`
- `callerDidDocument`
- local credentials
- Discovery proof
- request proof signed by the User Agent key

The request signature is Ed25519 over the canonical unsigned envelope hash.

## 6. Response Verification

The User Agent verifies the Service Agent response proof against the Service Agent DID Document.

The demo reports:

- `requestSignatureVerifiedByServiceAgent`
- `userCredentialVerifiedByServiceAgent`
- `responseSignatureVerifiedByUserAgent`
- `provenanceVerified`

The E2E script asserts all four checks.

## 7. MCP and A2A Compatibility

The current User Agent is an OpenAgentNet trusted invocation client. Full MCP and A2A client logic is future work.

Future work should:

- integrate trusted identity verification before MCP/A2A session setup
- attach signed identity context to MCP tool calls or A2A tasks
- verify signed task responses and callbacks
- support long-running and streaming task flows

## 8. Tests and Demo

The User Agent is exercised by:

```text
scripts/run-e2e-demo.ps1
```

The E2E demo validates:

- Discovery query
- signed invocation request creation
- local User Agent VC inclusion
- Service Agent verification of request and VC
- User Agent verification of signed Service Agent response

## 9. Next Work

- verify Discovery response signature directly in User Agent
- verify Service Agent DID Document against Root-anchored package data
- support VC expiration and revocation checks
- support negative tests for invalid signatures, missing VC, replayed nonce, and wrong target DID
- extract signing and verification logic into a reusable Python Agent Adapter package
